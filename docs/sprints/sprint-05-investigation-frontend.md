# Sprint-05 investigation: frontend match/scoring surface

Scope: map every frontend consumer of getMatchReport / previewDeepMatchScore / runDeepMatchScore, read the e2e journey end to end, check seed coupling, and flag gotchas. All findings CONFIRMED unless marked INFERRED or UNKNOWN. File:line pointers are relative to the repo root.

## 1. Frontend consumers

### Hooks

- `frontend/src/hooks/use-match-report.ts:5-13` -- `useMatchReport({resumeId, jobId})` wraps `api.getMatchReport` via the shared `useAsyncResource` (`frontend/src/hooks/_use-async-resource.ts`). Returns `{data, isLoading, error, refetch}`. No mutation, GET only.
- `frontend/src/hooks/use-deep-match-score.ts:24-73` -- `useDeepMatchScore()` exposes `preview(jobId, resumeNames)` -> `api.previewDeepMatchScore` and `run(jobId, resumeName)` -> `api.runDeepMatchScore`, each with its own `isPreviewing`/`isRunning`/`error` state. Errors are normalized to `MockApiError` (`frontend/src/lib/mock-api-error.ts`) and re-thrown so the caller can branch on `err.kind === "cap_reached"`.

### Screen: Match Explorer (`frontend/src/screens/match-explorer/index.tsx`)

- Route `/resumes/match-explorer` (`frontend/src/routes.ts:299-302`), lazy-loaded (`frontend/src/routes.ts:55`).
- Reads `resumeId`/`jobId` from search params, falling back to hardcoded fixtures `RESUME_ID_DISTRIBUTED` / `JOB_ID_STRIPE` (`match-explorer/index.tsx:38-39`, fixtures at `frontend/src/data/fixtures.ts:951`/`321`). So `page.goto("/resumes/match-explorer")` with **no query params** renders the canonical seeded pairing.
- `useMatchReport({resumeId, jobId})` drives the main render (`match-explorer/index.tsx:41-44`): loading -> `<Skeleton>` (`:146`), error -> `<ResourceError>` (`:149-151`), empty rubric -> `<EmptyState>` (`:154-159`), else renders three cards: Rubric (`h3` "Rubric" `:165`, one `<MatchRubricRow>` per `data.rubric` row), Gaps (`h3` "Gaps" `:175-178`, badges by severity + text), Strengths (`h3` "Strengths" `:203-206`).
- **previewDeepMatchScore trigger**: the "Re-run deep score" button (`Button` at `:137-139`, only rendered once `data` is loaded) calls `openDeepScore()` (`:54-63`), which opens `<CostPreviewDialog>` (`open=true`, `:232-245`) and calls `deepScore.preview(jobId, [resumeName])` where `resumeName` is the **hardcoded literal** `"Distributed-systems v4"` (`:52`) -- not derived from `data` or from the resume store.
- **runDeepMatchScore trigger**: `<CostPreviewDialog onConfirm={runDeepScore}>` (`:239`) -- the dialog's "Approve and run" button (default `confirmLabel`, `cost-preview-dialog.tsx:71`) calls `runDeepScore()` (`:65-88`), which calls `deepScore.run(jobId, resumeName)` (same hardcoded resume name) and on success shows a success toast with `result.score`/`result.costUsd` (`:69-72`); on `cap_reached` it flips `capReached=true` and the dialog re-renders its re-consent state (`cost-preview-dialog.tsx:74-99`, title "Monthly cap reached", button "Re-consent and continue").
- The MatchPill in the page header (`:136`) renders `data.score` with `kind="deep"` -- this pill reflects the **getMatchReport** score, not the result of a deep-score run (running deep-score does not refetch/replace `data`, it only toasts).
- "Generate tailored revision" button (`:220-226`) forks a resume draft (`forkResumeAsDraft`) and navigates to `/resume/:id/edit` carrying `data.gaps` in router state -- exercises `useResumeMutations`, tangential to match/score itself but part of the same screen.

### `data/api.ts` (HTTP adapter)

- `getMatchReport(args)` -- `frontend/src/data/api.ts:931-938`: `GET /match-report?resumeId=..&jobId=..`, thin pass-through, no wire transform.
- `previewDeepMatchScore(jobId, resumeNames)` -- `:1192-1225`: first calls `getResumes()` to build a name/id lookup, resolves each `resumeNames[i]` against `resumes` by exact id, then `startsWith`, then exact name (`:1197-1203`); posts `{resumeIds}` to `POST /jobs/{jobId}/preview-deep-score`; maps wire items into `CostPreviewItem[]` with a synthesized label `"Deep match score -- <resume name>"` (`:1214-1218`).
- `runDeepMatchScore(jobId, resumeName)` -- `:1227-1240`: resolves `resumeName` via `resolveResumeId()` (`:404-420`, same id/startsWith/name fallback chain, plus a numeric-index branch used elsewhere), posts `{resumeId}` to `POST /jobs/{jobId}/deep-score`, strips the wire-only `aiRun` telemetry field before returning.

### `data/types.ts` (app-facing types)

### `data/wire.ts`

### `data/fixtures.ts`

The screen's hardcoded `resumeName = "Distributed-systems v4"` (match-explorer/index.tsx:52) only resolves against the real resume because of the `startsWith` fallback in `api.ts` -- a fragile string match, see Gotchas.

## 2. Journey today (`frontend/e2e/core-journey.spec.ts`)

5. Assert `/jobs/:id` detail page renders via DB-backed `getJob` (`:144-146`).
6. Shortlist the created job from `/jobs` (click row -> "+ Shortlist" -> `POST /api/v1/shortlist` 201) and assert it lists on `/shortlist` (`:148-172`).
7. Sprint-04 3c extension: pull the seeded DEFAULT resume via `GET /api/v1/resumes` (`tag === "DEFAULT"`, expected to be `RESUME_ID_DISTRIBUTED`/"Distributed-systems") (`:185-199`), drive the application through `drafting -> applied` via `page.request.post(.../transitions)` carrying the bearer token pulled from `localStorage` (`authHeaders()`, `:64-75`), assert one illegal transition (`applied -> offer`) 422s with `kind: "invalid_transition"` (`:243-256`), assert the locked DEFAULT resume refuses DELETE with 409/`kind: "conflict"` both via API (`:272-278`) and via the Resumes screen UI (`:280-294`), and assert the submitted-resume snapshot is visible via API and UI (`:296-311`).

Conventions confirmed:

- **Selectors**: role-based (`getByRole("button", {name: /regex/i})`) and text-based (`getByText(...)`), no `data-testid` anywhere in this file or in the match/score components inspected (see Section 4 selector note).
- **Network assertion pattern**: `page.waitForResponse(predicate)` set up *before* the triggering click/goto, then awaited and status-checked -- used to prove a given screen is DB-backed, not mock-served.
- **Fresh-seed discriminator**: `RUN_TAG = Date.now()` baked into company/role strings so the test can assert "does not exist yet" without wiping the DB between runs (`:36-38`).
- **Auth**: real login form once per test file (this spec logs in itself, separately from `global-setup.ts`'s pre-fetch), reusing the demo tenant creds via `envVal("SEED_DEMO_EMAIL"/"SEED_DEMO_PASSWORD", ...)` (`frontend/e2e/auth.ts:12-34`) which reads `process.env` first, then the gitignored repo-root `.env`, then a hardcoded fallback (`wes.gilleland@gmail.com` / `employa-demo-1`) -- this is why the CI Playwright container (no `.env`) still works.
- **`page.request` bearer reuse**: `authHeaders()` (`:64-75`) pulls `access_token` straight out of `localStorage` (the same `TOKEN_KEY` `data/api.ts` uses) so raw API calls hit the DB-backed backend as the same tenant the UI is authenticated as -- this is the established pattern a match-score segment should reuse rather than re-logging in.

### Where a match-score segment would append

Natural insertion point: **after step 5** (job detail renders) and/or after step 7 (application reaches `applied`, DEFAULT resume is locked and known-good). At that point the test already has: an authenticated page, `defaultResume.id` (the seeded `RESUME_ID_DISTRIBUTED`), and `createdBody.jobId` (a **freshly created, non-seeded** job with no `match` data -- see Section 4 gotcha). Two realistic options:

- **Option A (cheapest, matches existing hardcoding)**: `page.goto("/resumes/match-explorer")` with no query params -- exercises the seeded canonical pairing (`JOB_ID_STRIPE` / `RESUME_ID_DISTRIBUTED`), which already has real match data in the DB (seeded `job.match`) once sprint-05 makes `getMatchReport` DB-backed. Assert on the `h3` headings ("Rubric"/"Gaps"/"Strengths"), rubric row count, then click "Re-run deep score" and assert the `CostPreviewDialog` renders itemized cost + "Approve and run", network-asserting `POST /jobs/{JOB_ID_STRIPE}/preview-deep-score` and `POST /jobs/{JOB_ID_STRIPE}/deep-score`, then assert the success toast.
- **Option B (exercises the newly-created entities)**: `page.goto(\`/resumes/match-explorer?resumeId=${defaultResume.id}&jobId=${createdBody.jobId}\`)`. This is more end-to-end but the freshly created job has no seeded match data (`match=None`, see Section 4), so `getMatchReport`'s post-sprint-05 DB-backed behavior for an *unscored* pair needs to be spec'd (empty rubric -> `EmptyState`? synthesize via the fake AI provider on first view? 404?) before a test can assert against it -- this is squarely an S3 spec decision, not something already answered in code.

## 3. Seed coupling

- `backend/app/scripts/seed.py` seeds, under one fixed demo user (`SEED_DEMO_EMAIL`/`SEED_DEMO_PASSWORD`): 7 demo jobs (`seed_demo_jobs`, `:72-100`, UPSERT-in-place by fixed UUID), demo shortlist entries (`seed_demo_shortlist`, `:103-117`), a minimal demo resume set (`seed_demo_resumes`, `:120-143`, UPSERT-skip), and 7 fixture applications with full stage-transition history (`seed_demo_applications`, `:159-238`, all-or-nothing).
- The 7 demo jobs and their fixed UUIDs (`JOB_ID_STRIPE` etc., `backend/app/store.py:1751`) and the demo resumes (`RESUME_ID_DISTRIBUTED` etc., `store.py:182`) come from `store.demo_job_seeds()` / `store.demo_resume_seeds()` -- **the same fixed UUIDs the frontend fixtures hardcode** (`frontend/src/data/fixtures.ts:321-327`, `:950-955`). This alignment is intentional and load-bearing: it's why `match-explorer`'s hardcoded fallback ids resolve to real seeded rows once match ops go DB-backed.
- `store.demo_job_seeds()`'s `JOB_ID_STRIPE` row **does** carry seeded `match` data (score 92, per `store.py` around `_seed_jobs()`/`_seed_jobs_inbox()`, consistent with `MATCH_REPORT_SCORE = 92` at `store.py:2641`). The DB `Job.match` column is JSONB with a CHECK constraint requiring `match IS NULL OR match->>'score'` to be a number (`backend/app/models.py:135-141`), and `job_mapper.py:53-55`/`:79` round-trips it.
- The **e2e-created job** (`core-journey.spec.ts` step 3) does NOT get seeded match data: `applications.py:317` sets `match=JobMatch(score=body.match, strengths=[], gaps=[])` where `body.match` comes from whatever the capture wizard's request body carries -- for a manually-typed capture with no fetch-derived score this is very likely `0`/unset (INFERRED -- did not trace the wizard's default `match` value through `add-app/index.tsx`, flagging as a follow-up if Option B above is chosen).
- **No shortlist/application entity hardcodes match-report ids** beyond the two constants above; `getMatchReport` (backend, current mock form) ignores `resumeId`/`jobId` validity entirely (see Gotchas), so today literally any UUID pair "works" against it.

## 4. Gotchas

1. **`getMatchReport` is presently mock-only and validates nothing.** `backend/app/api/routes/match.py:153-172` echoes whatever `resumeId`/`jobId` you pass onto a single hardcoded fixture (`store.MATCH_REPORT_SCORE/RUBRIC/GAPS/STRENGTHS`, `store.py:2641-2706`) with an explicit docstring calling this a "judgment call" that skips resume/job store validation. This is precisely the seam sprint-05 needs to close -- expect the AC to require it become DB-backed and probably to 404 on an unknown resume/job pair, matching the pattern `getJob` already uses.
2. **`previewDeepMatchScore`/`runDeepMatchScore` validate against the in-memory mock `store.jobs`, not the DB.** `match.py:76-77` and `:108-110` do `if id not in store.jobs: raise NotFoundError`. Today this "happens to work" for the 7 seeded fixture jobs only because `store.jobs`' fixed UUIDs and the DB seed's fixed UUIDs are the same constants (Section 3) -- but a **freshly created job (via the UI capture wizard) exists in the DB and NOT in `store.jobs`**, so `previewDeepMatchScore`/`runDeepMatchScore` against a newly-created job's id will 404 today. This is a second concrete instance of the same DB-vs-mock seam.
3. **Deep-score run result never feeds back into the rendered report.** `runDeepScore()` in `match-explorer/index.tsx:65-88` only toasts the new score/cost; it does not call `refetch()` on `useMatchReport`, so the rubric/gaps/strengths/pill on screen stay exactly as they were before the deep run. Any e2e assertion after "Approve and run" must assert on the **toast**, not on the rubric re-rendering.
4. **The resume passed to both deep-score calls is a hardcoded literal, resolved fuzzily.** `match-explorer/index.tsx:52` hardcodes `resumeName = "Distributed-systems v4"`; `api.ts`'s resolution chain (`:1197-1203`, `:404-420`) only matches it via a `name.startsWith(r.name)` fallback against the real resume name `"Distributed-systems"` (fixtures.ts:970-980 / store.py `RESUME_ID_DISTRIBUTED` block). Renaming that fixture resume, or any resume whose name is itself a prefix collision, would silently change which resume gets scored. Not exercised by any of the currently-*running* frontend tests (see next point).
5. **The two most relevant frontend unit tests are dead code, excluded from the vitest run.** `frontend/vitest.config.ts`'s `test.exclude` list retires `src/data/deep-match-score.test.ts` and `src/hooks/use-match-report.test.tsx` (along with 39 sibling files) as of the 2026-07-04 "seam-swap", with the comment: "these 41 files asserted on the mock data layer THROUGH api.ts, which is now a real HTTP adapter against the mock API backend -- in jsdom there is no backend, so they can only fail." The replacement is `src/data/adapter.integration.test.ts`, gated by `RUN_ADAPTER_IT=1` with the backend running -- **but that file currently has zero match/score coverage** (`grep -n "match" adapter.integration.test.ts` returns nothing). Net effect: there is presently **no automated coverage at all**, unit or integration, for `getMatchReport`/`previewDeepMatchScore`/`runDeepMatchScore` on the frontend side; the retired tests' `VITE_MOCK_FAIL` env-injection mechanism (`vi.stubEnv("VITE_MOCK_FAIL", "match-report:rate_limited")`, `use-match-report.test.tsx:28`) also has no implementation anywhere in current `api.ts` -- it was part of the pre-seam-swap client-side mock and is now inert. S3/S4 should plan to (re)add real coverage, likely in `adapter.integration.test.ts`, not resurrect the excluded files as-is.
6. **No in-app UI path reaches Match Explorer with a *specific* resume/job pair.** The Shortlist screen's `handleOpen()` (`frontend/src/screens/shortlist/index.tsx:66-73`) explicitly routes to job detail (`/jobs/:id`) for any entry that has a `jobId` -- match-explorer is only a fallback "for legacy entries that predate the jobId reference" (comment at `:63-65`, citing DEC-058/ADR-006: "Match stats live HERE [job-detail] (not on the match-explorer)"). The sidebar's "Resumes" nav item only *visually* marks itself active when on `match-explorer` (`frontend/src/components/shell/sidebar.tsx:305-311`) but its `<Link>` target is `/resumes`, not `/resumes/match-explorer`. The only other route in is the command palette (built from the static `ROUTES` table, `command-palette.tsx:49`/`:66`) or a direct URL. Practically: **an e2e match-score segment must `page.goto` the route directly** (Section 2, Option A/B) -- there is no click-path from the journey's existing steps (job capture -> shortlist -> applications) that lands on match-explorer with the created job.
7. **Job-detail's match panel is a completely separate data path from Match Explorer.** `frontend/src/components/domain/job-detail-view.tsx:182-238` renders `job.match.score/strengths/gaps` sourced from the `Job` entity itself (`GET /jobs/:id`, DB-backed already per sprint-02), not from `getMatchReport`/the deep-score endpoints. A newly-created job with no `match` set renders the "No resume scored against this job yet" empty copy (`:234-238`). Do not conflate this panel with the Match Explorer screen when writing ACs -- they are two independently-wired surfaces that happen to both show a "match score".
8. **Auth/error envelope on these calls is standard, nothing special.** All three ops go through the shared `call()` wrapper in `data/api.ts:128-186`: non-2xx responses are parsed into `{kind, path, message}` and converted via `httpErrorToMockApiError`; a `401` specifically clears the token and hard-redirects to `/login` (`:172-178`), a `403` does not. `getMatchReport` surfaces its error through `<ResourceError>` (retry button wired to `refetch`, `match-explorer/index.tsx:149-151`); the deep-score calls surface their error through `<CostPreviewDialog error={...} onRetry={openDeepScore}>` (`:237-238`) except when the error is specifically `cap_reached`, which is diverted to the dedicated re-consent UI instead of the generic error panel (`:73-88` in `cost-preview-dialog.tsx`).
9. **`runDeepMatchScore`'s cap enforcement is server-side-only and pre-spend.** `backend/app/api/routes/match.py:112-114` checks `unit_cost > store.cap_remaining_usd()` and raises `CapReachedError` (402, presumably `kind: "cap_reached"`) *before* incrementing `store.month_spend_usd` (`:128`) -- so a capped run never partially charges. `store.month_spend_usd`/`cap_remaining_usd()` (`store.py:1691-1723`, `:2713-2718`) are in-memory module globals, meaning **spend resets on backend process restart** and is not tenant-scoped -- worth flagging for the S3 spec since DB-backing match ops should presumably also DB-back (or explicitly punt on) the monthly-cap ledger.
10. **`AiRunEnvelope`/`synthetic=True` already exists as the "fake AI provider" marker.** `match.py:130-140` builds an `AiRunEnvelope(provider="fake", model=store.DEEP_MATCH_SCORE_MODEL, ..., synthetic=True)` and attaches it to the wire response as `aiRun`, which `frontend/src/data/api.ts:1236-1238` explicitly strips before returning `DeepMatchResult` to callers ("aiRun telemetry is dropped -- the hook's return type is the bare result"). This looks like a pre-existing hook sprint-05 can extend (real provider seam) rather than a new concept to invent -- but note the frontend currently throws this telemetry away entirely, so if the AC wants the UI to show provider/synthetic info, `api.ts`/`types.ts`/the screen all need changes, not just the backend.

## Summary for the caller

Wrote the full report to `docs/sprints/sprint-05-investigation-frontend.md`. Highlights below; see the doc for file:line detail.

- Frontend surface: `useMatchReport`/`useDeepMatchScore` hooks, `match-explorer/index.tsx` screen (route `/resumes/match-explorer`), `data/api.ts` (getMatchReport/previewDeepMatchScore/runDeepMatchScore), `data/types.ts` (MatchReport/CostPreview/DeepMatchResult) -- no `wire.ts` involvement, no `data-testid`s anywhere (role/text selectors only).
- **Big gotcha**: no in-app click-path reaches match-explorer with a specific resume/job pair -- Shortlist explicitly routes elsewhere per DEC-058; an e2e segment must `page.goto` the route directly, either with no query params (seeded canonical pair) or with the created job/resume ids (requires an S3 decision on unscored-pair behavior).
- **Big gotcha**: current backend match ops are two different levels of "mock": `getMatchReport` ignores ids entirely; `preview/runDeepMatchScore` validate against in-memory `store.jobs`, which 404s for any job created through the UI capture flow (DB-only). Both are exactly the seam sprint-05 needs to close.
- **Coverage gap**: the two frontend tests that once covered this surface (`deep-match-score.test.ts`, `use-match-report.test.tsx`) are excluded from the vitest run since the 2026-07-04 seam-swap and reference a `VITE_MOCK_FAIL` mechanism that no longer exists in `api.ts`. The designated replacement, `adapter.integration.test.ts`, has zero match/score coverage today.
- Seed alignment is real and load-bearing: frontend's hardcoded `JOB_ID_STRIPE`/`RESUME_ID_DISTRIBUTED` fixtures share fixed UUIDs with `backend/app/store.py` and `seed.py`'s DB seed, which is why the current hardcoded fallback "just works" once DB-backing lands for the seeded pair.
- Deep-score run does not refetch the report (toast-only feedback); the resume name passed to preview/run is a hardcoded literal resolved via fuzzy `startsWith` matching, not an id.
