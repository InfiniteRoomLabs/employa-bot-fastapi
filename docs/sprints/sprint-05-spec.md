## PINs (spec-frozen decisions; D1 finding in parens)

- PIN-A1 (append-only shape, D1-9 + the S0 rubric warning): `ai_run`, `ai_run_event`, and `match_report` are all append-only via `enforce_append_only()` AND INSERT-revoked from app_runtime (SELECT-only, forge-proof -- the PIN-19 posture). NO mutation function exists for any of them: they are write-once rows inserted INSIDE the two budget functions (the resume_snapshot lesson: plain INSERT riding a guarded operation's transaction). The ONLY new SQL functions are `ai_reserve_run` and `ai_settle_run`, justified solely by the budget row's locked-row arithmetic -- the one genuinely mutable, contention-prone row. The sprint-04 stage-mutation pattern is deliberately NOT copied onto the write-once tables.
- PIN-A2 (run status is event-derived, per D1's design answer): ai_run carries NO status column. Status derives from ai_run_event: reserved event only = "reserved" (open); succeeded/failed terminal event = terminal. v3's "status CHECK" lands as the CHECK on ai_run_event.kind. Wire AiRunStatus "cancelled" is never produced in 0.1 (no cancel path exists); recorded, not pretended away.
- PIN-A3 (idempotency, D1-2/D1-3): the contract has NO client idempotency key (RunDeepScoreBody = {resumeId} only -- CONFIRMED). Key is server-derived INSIDE ai_reserve_run under the budget row lock: `deep_match:{job_id}:{resume_id}:{attempt_seq}` where attempt_seq = count of SETTLED (terminal) runs for the triple. UNIQUE(user_id, idempotency_key) on ai_run is the DB backstop. Adoption rule: if the computed key already names an existing run (necessarily open, by construction), the function returns that run with adopted=true and does NOT reserve again -- so a retry after any post-reservation failure resumes the open run (conjunct 5), while a re-run after settlement gets a fresh key/run/report-version (legitimate re-scoring).
- PIN-A4 (retry state machine, D1-2/D1-8): the binding invariant, v3 Phase B item 4 VERBATIM: "idempotency key + unique constraint so a retry after a DB failure returns the existing run (no double 'charge')". The invariant's stated scope is DB failure. Every DB-failure boundary is covered: (i) crash after reserve-commit, before provider: retry adopts the open run, calls the provider, settles -- one reservation, one charge. (ii) provider raises: route settles failed (releases the reservation) and maps to 503 provider_unavailable; a later request is a NEW run. (iii) crash after provider success, before settle -- INCLUDING a DB failure during the settle transaction itself (settle rolls back atomically, the run stays open): retry adopts, RE-INVOKES the fake provider (deterministic + in-process + zero real cost: re-invocation is safe by construction; asserted with an invocation counter), settles once -- the partial-unique terminal-event index makes double-settle impossible at the DB. (iv) HTTP response lost AFTER a successful settlement commit: not a DB failure -- the DB flow completed; this is network-layer loss, outside the invariant's stated scope, and no server-side mechanism expressible under the frozen contract can distinguish that retry from a deliberate re-run (the contract's RunDeepScoreBody carries no client idempotency key, the op is synchronous, and adding a key is an mvp-api.yaml edit = a standing STOP condition for this phase). The maximal invariant implementable under the frozen contract is exactly (i)-(iii); (iv) is recorded as the idempotency boundary (reservation-to-settlement) with the real-provider requirement (provider-side idempotency/reconciliation before any non-fake provider ships) a BINDING entry criterion for the Phase-C contract revision, carried on the debt ledger at S7.
- PIN-A5 (reservation arithmetic, D1-3/D1-4): ai_reserve_run runs in its own SHORT transaction, committed before any provider work: lock the current-month budget row (upsert-then-SELECT FOR UPDATE), adoption check, then the guarded UPDATE `SET reserved_usd = reserved_usd + :max WHERE ... AND spent_usd + reserved_usd + :max <= cap_usd` -- zero rows = SQLSTATE EMP30 -> 402 cap_reached, nothing inserted. The DB CHECK `spent_usd + reserved_usd <= cap_usd` is the belt under the function's suspenders. The provider is called with NO transaction open; ai_settle_run is a second short transaction. A blocking-provider test proves a second connection sees the COMMITTED reservation while the provider hangs (conversion is outside and after, structurally).
- PIN-A6 (settlement liveness, D1-1): ai_settle_run(succeeded): budget.reserved_usd -= run.reserved_max; spent_usd += actual (CHECK actual <= reserved_max); insert succeeded event + match_report(version = next). ai_settle_run(failed): reserved_usd -= reserved_max (full release), insert failed event, NO report. Post-settle the headroom is available again: ACs assert exact balances after success, after failure, and that a subsequent run succeeds.
- PIN-A7 (current pointer, D1-6): the current match_report version for a triple is MAX(version) -- derived, no mutable pointer state, unambiguous by construction. getMatchReport = ORDER BY version DESC LIMIT 1. Versions are immutable rows (append-only table); UNIQUE(user_id, job_id, resume_id, version) makes the sequence collision-free (version assigned inside ai_settle_run under the budget lock).
- PIN-A8 (fake provider determinism, D1-13): FakeAiProvider.deep_match_score(input) is a pure function of its input snapshot (no DB, no clock, no randomness): score = min(99, baseline + 3) when the job carries a match baseline, else 80; strengths/gaps from the baseline when present, else the canned pair; rubric = the canonical 4-row fixture; actual_cost = the unit cost passed in. Identical across repeated calls and fresh instances (tested). synthetic := (provider == "fake"), derived at the mapper.
- PIN-A9 (explicit-only factory, D1-13): app/ai/factory.py holds the ONLY provider construction site: `get_provider(settings)` returns FakeAiProvider iff AI_PROVIDER == "fake", else raises ProviderUnavailableError (503) -- unknown, empty, or any other value. No fallback path exists anywhere (grep-able single site). Settings declares `AI_PROVIDER: str = "fake"` -- a declared configuration default for the only provider that exists in 0.1, not a runtime fallback; compose.yml sets it explicitly for the stack and CI.
- PIN-A10 (throttle, manifest rate-limited tag): runDeepMatchScore is throttled per-user BEFORE the reservation (no spend on throttled requests) via the sprint-01 throttle module pattern; `DEEP_SCORE_THROTTLE_USER_PER_MINUTE: int = 10` in Settings. ## Schema (migration, one revision; every binding convention explicit)

All four tables: `user_id uuid NOT NULL REFERENCES "user"(id) ON DELETE CASCADE`, `UNIQUE (user_id, id)` composite anchor, ENABLE + FORCE ROW LEVEL SECURITY with the standard `app.user_id` GUC policy (exemplar 4c17ea8b5656 / 5f1fb22cd505), partial index on user_id, `schema_version integer NOT NULL DEFAULT 1 CHECK (schema_version >= 1)`, all timestamps timestamptz, all money NUMERIC(10,6). RLS/grants/append-only are migration-only artifacts (DEBT-6: hand-strip autogenerate).

- `user_ai_budget`: id uuid PK; month_start date NOT NULL; cap_usd NUMERIC(10,6) NOT NULL CHECK (cap_usd > 0); spent_usd NUMERIC(10,6) NOT NULL DEFAULT 0 CHECK (spent_usd >= 0); reserved_usd NUMERIC(10,6) NOT NULL DEFAULT 0 CHECK (reserved_usd >= 0); CHECK (spent_usd + reserved_usd <= cap_usd) [ck_budget_cap -- the arithmetic belt]; UNIQUE (user_id, month_start); created_at/updated_at timestamptz NOT NULL. Grants: app_runtime SELECT only -- INSERT/UPDATE happen inside the SECURITY DEFINER functions; DELETE only via user cascade. NOT append-only (the one mutable row, mutated only by the two functions).
- `ai_run` (append-only, SELECT-only): id uuid PK; job_id uuid NOT NULL, CONSTRAINT fk_ai_run_job FOREIGN KEY (user_id, job_id) REFERENCES job (user_id, id); resume_id uuid NOT NULL, CONSTRAINT fk_ai_run_resume FOREIGN KEY (user_id, resume_id) REFERENCES resume (user_id, id); budget_id uuid NOT NULL, CONSTRAINT fk_ai_run_budget FOREIGN KEY (user_id, budget_id) REFERENCES user_ai_budget (user_id, id); kind text NOT NULL CHECK (kind IN ('deep_match_score')); provider text NOT NULL; model text NOT NULL; reserved_max_usd NUMERIC(10,6) NOT NULL CHECK (reserved_max_usd > 0); idempotency_key text NOT NULL; UNIQUE (user_id, idempotency_key) [uq_ai_run_idempotency]; created_at timestamptz NOT NULL DEFAULT now().
- `ai_run_event` (append-only, SELECT-only): id uuid PK; run_id uuid NOT NULL, CONSTRAINT fk_ai_run_event_run FOREIGN KEY (user_id, run_id) REFERENCES ai_run (user_id, id); kind text NOT NULL CHECK (kind IN ('reserved','succeeded','failed')); actual_cost_usd NUMERIC(10,6) NULL CHECK ((kind = 'succeeded') = (actual_cost_usd IS NOT NULL)) [ck_event_cost_shape]; created_at timestamptz NOT NULL DEFAULT now(); partial UNIQUE INDEX uq_ai_run_event_reserved ON (user_id, run_id) WHERE kind = 'reserved'; partial UNIQUE INDEX uq_ai_run_event_terminal ON (user_id, run_id) WHERE kind IN ('succeeded','failed').
- `match_report` (append-only, SELECT-only): id uuid PK; job_id uuid NOT NULL + fk_match_report_job -> job(user_id, id); resume_id uuid NOT NULL + fk_match_report_resume -> resume(user_id, id); ai_run_id uuid NOT NULL + fk_match_report_run -> ai_run(user_id, id) + UNIQUE (user_id, ai_run_id) [one report per run]; version integer NOT NULL CHECK (version >= 1); UNIQUE (user_id, job_id, resume_id, version) [uq_match_report_version]; score integer NOT NULL CHECK (score BETWEEN 0 AND 100); rubric jsonb NOT NULL, named CHECK ck_match_report_rubric ((jsonb_typeof(rubric) = 'array') IS TRUE); gaps jsonb NOT NULL, named CHECK array; strengths jsonb NOT NULL, named CHECK array; created_at timestamptz NOT NULL DEFAULT now().
- Functions `ai_reserve_run(p_job_id, p_resume_id, p_kind, p_provider, p_model, p_max_usd, p_cap_usd)` and `ai_settle_run(p_run_id, p_outcome, p_actual_usd, p_score, p_rubric, p_gaps, p_strengths)`: owner-owned SECURITY DEFINER, `SET search_path = public, pg_temp`, EXECUTE REVOKEd from PUBLIC and GRANTed to app_runtime; tenant from `current_setting('app.user_id', true)` (raise EMP01-style on unset -- fail closed); explicit user_id predicates everywhere (SECURITY DEFINER bypasses RLS). SQLSTATEs (stage_flow.py taxonomy extended): EMP30 cap_reached; EMP31 run not found / not owned (route maps 404); EMP32 run already settled (route treats as adopted-terminal: read the report and return it); EMP33 ownership failure on job/resume (404, tenant-indistinguishable); EMP34 settle payload illegal (defensive, 500-class). ai_reserve_run returns (run_id, adopted bool, reserved_max, budget snapshot); ai_settle_run returns (event_id, report_id, version, created_at).

## runDeepMatchScore route flow (normative)

1. Throttle check (PIN-A10) -> 429 before anything.
2. Txn 1 (tenant session): SELECT job (ownership + match baseline) -> 404; call ai_reserve_run -> EMP30 -> 402 cap_reached, EMP33 -> 404. Commit. Capture (run_id, adopted, reserved_max, run created_at).
3. Provider OUTSIDE txn: get_provider(settings) (PIN-A9) -> FakeAiProvider.deep_match_score(DeepMatchInput(job baseline fields, resume_id, unit_cost)) -> deterministic result (PIN-A8). On ANY provider exception: txn: ai_settle_run(failed) (releases reservation, PIN-A6), then raise ProviderUnavailableError (503).
4. Txn 2: ai_settle_run(succeeded, actual, score, rubric, gaps, strengths); EMP32 (a concurrent retry settled first) -> read the existing report for the run and serve it (identical by determinism). Build the ENTIRE wire response before commit (post-commit lazy-load trap, resumes.py:22 precedent).
5. Response: DeepMatchResult(jobId, resumeId, score, kind=deep, strengths, gaps, costUsd=actual, aiRun=AiRunEnvelope(provider, model, status=succeeded, startedAt=run.created_at, finishedAt=event.created_at, durationMs=delta, estimatedCostUsd=reserved_max, actualCostUsd=actual, synthetic=True)).
